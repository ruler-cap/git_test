# 上层移动操作模型与底层运控结合：论文导向调研

调研日期：2026-07-13

## 范围与结论

本调研只回答一个接口问题：视觉/语言/模仿学习等**上层移动操作模型**应输出什么，才能由**底层轮式全身控制器**将底盘、躯干、双臂和手部作为一个整体执行。这里的“底层”可为 QP/WBIK、MPC/WBC 或学习式全身跟踪策略。

论文证据表明，最有效的解耦点是**末端任务空间**，而不是底盘命令和关节命令的拼接：上层给出 6D 末端关键帧、相对增量、可供性点偏移或任务坐标系轨迹，底层根据当前底盘/关节状态和约束求全身可行解。HoMeR（轮式全向底盘）是与此最直接对应的工作；其上层按阶段切换绝对末端位姿和相对末端增量，WBC 同时生成底盘与手臂命令。

对于高动态轮式人形，建议保持相同的上层接口，但将底层从运动学 WBIK 升级为动力学 MPC/WBC，并把稳定性、轮胎摩擦、力矩和接触约束放入优化器。上层不应直接学习这些约束。

## 1. 论文中已验证的结合方式

| 工作 | 平台与发表时间 | 上层/中层输出 | 底层执行与耦合方式 | 对接口设计的直接启示 |
|---|---|---|---|---|
| **HoMeR** | 全向轮底盘 + 7-DoF 手臂，arXiv 2025-06 | 远场为绝对 6D EE keypose；近场为相对 6D EE delta；另输出夹爪与模式 | 快速运动学 WBC 将 EE 命令映射为底盘与手臂的协调命令 | 最直接支持“世界/物体系关键帧 + 相对精操作”的混合接口 |
| **Mobile ALOHA** | 轮式移动双臂，CoRL 2024 | 底盘与双臂的全身遥操作轨迹，行为克隆直接预测平台动作 | 原有底盘/关节伺服执行，没有显式 WBC 分配层 | “底盘偏移 + 关节动作”可运行，但高度具身相关，适合作为基线 |
| **Zero-Cost Whole-Body Teleoperation / MoMa-Teleop** | 轮式移动操作臂，2024 preprint，后续 RA-L 接收 | 操作者只给 6-DoF EE 信号；也可用 EE-only 的示范学习任务 | RL base agent 根据 EE 意图生成支持性底盘运动，IK 生成上身关节 | 可只采 EE 数据，而把底盘协同交给独立的全身/底盘模块 |

| **Reactive Base Control in Dynamic Environments** | 轮式移动操作，2023 | 当前/下一操作目标和动态障碍感知 | 反应式底盘控制 + QP 冗余分解；同时约束底盘、手臂避障 | 底层需要实时可行性和障碍反馈，不能仅复现离线全身轨迹 |
| **HoMMI** | RB-Y1 全向轮、6-DoF 躯干、双 7-DoF 臂，RSS 2026 | 双手 EE 位姿与头部 look-at point；观测和动作均表示在左夹爪中心帧 | 约束感知 WBC；CoM 稳定优先级高于双手跟踪，并调节底盘/躯干/手臂权重 | 对轮式人形，推荐“夹爪中心任务帧 + 双手相对约束 + 视线点”而非全身关节动作 |
| **UMI on Legs** | 带臂四足，CoRL Workshop 2024 | 任务坐标系中的 EE 轨迹 | 仿真训练的全身控制器跟踪轨迹 | 虽非轮式，但证明任务轨迹数据与具身 WBC 可解耦；适合跨平台数据集 |
| **Hierarchical Optimization for WIP Humanoids** | 双轮倒立人形，2018 | 高层 CoM 目标及操作空间任务 | 下层短时域全身优化同时处理车轮动态与关节操作 | 高动态双轮平台需显式向底层传递 CoM/稳定性目标，不能只有 EE 位姿 |

## 2. 每篇工作的具体接口与方法

### 2.1 HoMeR：最贴近“上层模型 + 轮式 WBC”

Sundaresan et al. 将上层模仿学习策略限定在低维末端空间，底层使用运动学 WBC。平台为**全向轮移动底盘加 7-DoF 手臂**，实机任务包括开柜门、扫垃圾和整理靠垫。

- **远场 keypose policy**：第三视角点云中预测任务相关显著点、从该点到目标位置的偏移、末端方向、夹爪状态及下一控制模式。它产生绝对 6D 末端目标，适合接近对象。
- **近场 dense policy**：使用第三视角和腕部 RGB，输出相对 6D 末端位姿增量，适合接触前的精细调整。
- **WBC**：将上述末端命令转为移动底盘与手臂的协调关节命令。论文附录给出其 IK 任务包括 6D 末端位姿、关节姿态正则和底盘速度阻尼，并有速度/关节位置约束。
- **结果**：三项仿真和三项真实家庭任务的总体成功率 79.17%，仅用每任务 20 条示范；文中对比显示，移除混合动作或 WBC 都会显著降低成功率。

它对应本问题的最佳范式：上层无需预测底盘如何转向或手臂各关节如何避奇异；它只决定“手要到哪里、如何微调、何时抓取”，WBC 决定“轮子、躯干和手臂怎样共同到达”。

论文：[arXiv](https://arxiv.org/abs/2506.01185)，[项目与代码](https://homer-manip.github.io/)，发表时间：2025-06。

### 2.2 Mobile ALOHA：全身端到端动作的可行基线

Fu、Zhao、Finn 在 Mobile ALOHA 中通过移动底盘与双臂的全身遥操作采集数据，并以监督行为克隆学习移动双臂任务，如做菜、开双门柜、进电梯和清洗。

- **动作/数据特点**：系统保留了底盘和双臂的直接控制链路；该路径可视作“底盘命令 + 双臂/夹爪关节命令”式数据。
- **优势**：工程上最简单，遥操作数据和部署动作同构；仓库提供 HDF5 数据采集、可视化和回放。
- **接口局限**：上层策略必须自行学会底盘-手臂冗余分配、工作空间边界和平台几何。因此更换底盘、臂长或加入躯干自由度时，动作空间和数据分布都会变化；也无法天然保证高动态稳定约束。

论文：[arXiv](https://arxiv.org/abs/2401.02117)，[项目](https://mobile-aloha.github.io/)，[代码](https://github.com/MarkFzp/mobile-aloha)，发表时间：2024-01；CoRL 2024。

### 2.3 MoMa-Teleop：6D 末端输入，学习式底盘协同

Honerkamp et al. 的 MoMa-Teleop 提供了一个强工程证据：用户只操作 **6D 末端信号**，其余全身自由度由模块化后端补全。

- **输入链路**：任意 6D 操作信号（手引导或摇杆）先被转为 EE 速度/运动；RL base agent 根据期望 EE 运动产生支持性底盘运动；IK 解出机械臂和躯干关节命令。
- **数据对比**：作者明确比较 whole-body 数据（共同拟合 EE 与底盘位姿）和 EE-only 数据（仅建模末端与夹爪，再用同一底盘 agent 实现协同）。EE-only 在底盘初始位置变化和新障碍条件下泛化更好。
- **意义**：证明底盘轨迹不必成为上层数据标签。对你关心的轮式人形，这意味着可以单独采集“人或末端”操作数据，而将底盘补偿交给 WBC/RL 控制器。

论文：[arXiv](https://arxiv.org/abs/2409.15095)，发表时间：2024-09；作者公开材料显示该工作后续被 RA-L 接收。项目代码状态请以论文项目页为准。

### 2.4 Manipulation On-The-Move：把“下一任务目标”传入底盘控制

Burgess-Limerick et al. 关注机器人在移动过程中完成抓放，而不是先停车再操作。系统的移动底盘控制器读取**当前机器人位姿、当前操作目标、下一目标**，输出底盘期望速度并估计何时进入操作范围；机械臂采用反应式控制完成当前抓取。

- 真实系统在 120 次试验中仅失败 1 次，报告相对基线的任务时间最多减少 48%。
- 该工作不是学习模型，但补充了一个常被忽略的上层接口字段：除了 EE/对象目标，上层还应给出任务队列中的下一目标或进度，使底盘在精操作时也能朝下一地点移动。
- 后续工作将动态障碍物纳入底盘控制，并在 QP 冗余分解中加入手臂避障约束。

论文：[On-The-Move 架构，arXiv](https://arxiv.org/abs/2212.06991)，发表时间：2022-12 / ICRA 2023；[动态环境底盘控制，arXiv](https://arxiv.org/abs/2309.09393)，发表时间：2023-09。

### 2.5 HoMMI：轮式人形的双手、视线与稳定性接口

Xu et al. 在 RB-Y1（全向底盘、6-DoF 躯干、双 7-DoF 手臂、2-DoF 颈部）上学习长程移动双臂操作。它不是让模型预测全身关节，而是使用跨具身 hand-eye 策略与约束感知 WBC。

- **坐标系选择**：将夹爪本体状态、夹爪动作、头部 pointmap 和头部 look-at point 都变换到**左夹爪中心帧**，降低头部运动与不同身体尺寸导致的坐标漂移和具身差异。
- **上层输出**：双手末端位姿和头部 look-at point。
- **WBC**：在优化层级中，把 CoM 位于底盘支撑范围内置于最高优先级，其后才是双手末端精确跟踪；底盘、躯干和手臂姿态的相对权重随任务调整。
- **意义**：对双臂轮式人形，末端动作结构应加上双手相对约束和主动视觉目标；在动态平台上，稳定性不是上层模型应隐式学习的“副作用”，而是 WBC 的硬/高优先级约束。

论文：[arXiv](https://arxiv.org/abs/2603.03243)，[项目](https://hommi-robot.github.io/)，[代码](https://github.com/xxm19/hommi)，发表时间：2026-03；RSS 2026。

### 2.6 UMI on Legs 与双轮倒立人形：可迁移接口和高动态补充

UMI on Legs 的高层视觉策略只输出**任务坐标系中的 EE 轨迹**，而仿真训练的低层全身控制器跟踪该轨迹。该分离使来自手持 UMI 的真实世界操作数据能与机器人仿真控制数据组合。虽然平台为四足带臂，不是轮式，但这是“上层任务数据可跨具身、下层 WBC 与机器人绑定”的直接论据。

Zafar et al. 针对双轮倒立人形采用分层优化：高层基于简化模型给出 CoM 目标，下层在短时域内利用更完整动力学规划车轮与全身关节。这说明对双轮动态平衡平台，仅给末端任务还不足够；WBC/MPC 必须内部维护或从上层获得 CoM/稳定性参考。

论文：[UMI on Legs，arXiv](https://arxiv.org/abs/2407.10353)，发表时间：2024-07；[WIP Humanoid，arXiv](https://arxiv.org/abs/1810.03074)，发表时间：2018-10。

## 3. 五种数据结构：对应论文、字段和评价

下表将问题中的五种表示与上述论文逐一对应。`W` 为世界系，`B` 为底盘系，`O` 为对象/任务系，`E` 为末端 TCP；`T_A_B` 表示从 `B` 到 `A` 的 SE(3) 位姿。

| 表示 | 最直接论文证据 | 推荐字段 | 适用阶段 | 判断 |
|---|---|---|---|---|
| 1）末端世界坐标系 | HoMeR 远场 keypose；On-The-Move 的对象/目标状态 | `T_W_E_des`、`gripper`、`mode`、`horizon` | 远距离接近、全局关键帧 | 必需但不宜单独承担精操作；受 SLAM/物体定位漂移影响 |
| 2）末端相对坐标系 | HoMeR dense policy；UMI on Legs 任务帧轨迹；HoMMI 夹爪中心帧 | `frame_id`、`Delta T_E_E` 或 `T_O_E`、`dt`、`gripper` | 近场、接触前、视觉闭环 | **首选主接口**；一定要显式记录参考坐标系 |
| 3）末端偏移量 | HoMeR 的显著点 + offset | `p_anchor`、`Delta p`、`R_E_des`、`anchor_frame`、夹爪/模式 | 把手、孔、按钮、抓取可供性 | 极适合视觉/语言模型；偏移不能替代完整 6D 姿态 |
| 4）底盘偏移量 + 关节坐标 | Mobile ALOHA；whole-body GMM 基线于 MoMa-Teleop | `v, omega` 或 `Delta x_B, Delta y_B, Delta yaw_B`，以及 `q_des`/`qdot_des` | 同平台行为克隆和遥操作 | 可部署但具身强绑定，WBC 协同被上层隐式承担 |
| 5）全身关节端到端 | Mobile ALOHA 的直接全身控制思路；HoMMI/WIP 的低层执行端 | `wheel_cmd, q_des_all` 或 `tau_all`，并记录状态与安全边界 | 已训练的低层技能、动作重定向 | 只应作为低层输出/训练标签；不建议作为跨任务上层接口 |

### 3.1 模式 1：世界坐标系末端位姿

最小动作包：`{frame: W, T_W_E_des, gripper, mode, horizon_s}`。它适合让上层表达“到柜门前方的预抓取姿态”。HoMeR 的远场策略预测绝对 6D 位姿；但其论文同时引入了近场相对控制，说明单纯世界系并不足以解决高精度接触。

对移动机器人，世界系目标必须绑定定位质量。若 `T_W_B` 有漂移，则 `T_W_E_des` 的误差会直接传给末端。工程上应在接近目标后切换到对象系/腕部视觉相对控制，而不是持续追踪远场世界系姿态。

### 3.2 模式 2：末端相对位姿/任务坐标系

建议优先使用：`{frame: O or E, delta_twist_6d, dt, gripper, contact_mode}`。若有稳定对象或操作面，使用 `T_O_E_des`；若物体坐标系无法稳定估计，则使用当前 TCP 增量 `Delta T_E_E_next` 并由视觉闭环不断修正。

HoMeR 的绝对/相对混合与 UMI on Legs 的任务帧轨迹都支持这一选择。关键点是末端位姿用 SE(3) 复合：`T_next = T_current * Exp(xi_hat * dt)`，不能把旋转和位移作为普通向量长期累加。旋转应使用 6D rotation 或单位四元数，避免欧拉角奇异和跳变。

### 3.3 模式 3：以可供性点为锚的偏移

建议结构：`{anchor_frame, p_anchor, delta_position, approach_axis or R_E_des, gripper, mode}`。HoMeR 的 keypose policy 先在点云中找显著点，例如把手，再预测该点到最终末端目标的位置偏移，同时单独预测方向、夹爪和模式。

这种结构尤其适合 VLA：语言模型可选“哪个点/物体”，视觉模块定位点，策略只预测局部几何偏移。注意不能只保留三维偏移；插孔、开门和抓取仍需要末端朝向、接近方向和夹爪状态。

### 3.4 模式 4：底盘偏移 + 关节坐标

推荐记录形式为：`{base: [v, omega] (非完整底盘) or [vx, vy, omega] (全向底盘), q_torso_des, q_arm_des, q_hand_des}`。Mobile ALOHA 证明其可用于真实家庭任务，但这类策略的输出维度随机器人结构变化，无法直接转移到另一种底盘或双臂/躯干组合。

因此它的正确定位是：快速建基线、遥操作回放、训练同构低层策略。对于“底盘和上身统一参与运控”，它不如模式 2，因为协同解算发生在学习模型内部而不是可检查的 WBC 优化器中。

### 3.5 模式 5：全身关节端到端

形式可以是 `{wheel_cmd, q_des_all}`、`{wheel_cmd, qdot_des_all}` 或 `{tau_wheel, tau_joint}`。它保留了全部运动细节，但使高层模型必须学习轮式非完整约束、底盘动力学、奇异位形、碰撞与稳定性，数据量和同步要求都很高。

HoMMI 和 WIP 工作的共同启示是：全身关节/力矩应由具有约束感知的底层控制器生成；即便使用端到端模型，也应在其后设置速度、力矩、碰撞、CoM/ZMP 和接触安全过滤。该表示不宜作为跨任务、跨平台移动操作模型的主动作空间。

## 4. 面向轮式人形的推荐接口

### 4.1 上层到 WBC 的动作包

```text
TaskReference {
  timestamp, horizon_s, mode,
  ee_left:  {frame_id, pose_or_delta_6d, gripper, contact_mode},
  ee_right: {frame_id, pose_or_delta_6d, gripper, contact_mode},
  bimanual_relative_constraint: T_Eleft_Eright_des (optional),
  gaze_target: p_O_or_W (optional),
  object_anchor: {id, pose/confidence} (optional),
  next_task_target: {id, pose} (optional),
  confidence
}
```

- **远场**：`mode = keypose`，使用 `T_W_E` 或 `T_O_E`。参考 HoMeR 与 On-The-Move。
- **近场高精度**：`mode = relative`，使用任务系的 `Delta T`；接触时增加期望法向力、阻抗或导纳参数。参考 HoMeR 和 MoMa-Teleop 的 EE 驱动思想。
- **双臂与主动视觉**：附加 `T_Eleft_Eright` 和 `gaze_target`，参考 HoMMI；这比独立预测两只手的 6D 动作更能保持共同搬运或双手抓持的一致性。
- **任务连续移动**：附加 `next_task_target`，参考 On-The-Move；底盘可在当前抓放尚未结束时开始为下一目标布局。

### 4.2 WBC/MPC 的责任边界

WBC 输入 `TaskReference`、机器人状态、地图/距离场、力触觉与底盘状态，输出 `wheel_cmd`、`qdot/qddot/tau`。它必须负责：

1. 底盘运动学约束：差速/阿克曼底盘只能接收可行的 `v, omega`；全向底盘才可用 `vx, vy, omega`。
2. 全身分配：以 EE 误差为主任务，让底盘、躯干与双臂共同贡献雅可比；同时保留舒适姿态和视线等次任务。
3. 安全约束：关节、速度、力矩、轮胎摩擦、自碰/环境碰撞；对双轮倒立或高重心平台还应加入 CoM、倾覆/ZMP。
4. 反馈：返回 `feasible`、EE 跟踪误差、碰撞/稳定裕度、接触状态和拒绝原因。上层可据此重定位可供性点、减速、移动底盘或触发恢复。

## 5. 选型结论

| 需求 | 应采用的数据结构 | 最相关论文 |
|---|---|---|
| 轮式机器人边走边抓放 | 世界/对象系 keypose + 当前/下一目标 | HoMeR；Reactive Mobile Manipulation On-The-Move |
| 末端高精度、视觉闭环 | 任务/夹爪系相对 6D delta，必要时加接触参数 | HoMeR；MoMa-Teleop；UMI on Legs |
| 把手、按钮、孔等语义目标 | 显著点/可供性点 + 6D offset 与朝向 | HoMeR |
| 双臂轮式人形与主动视角 | 双手任务帧轨迹 + 手间相对约束 + look-at point | HoMMI |
| 高动态双轮或重载 | 同上层 EE 轨迹，但由动力学 MPC/WBC 加 CoM/稳定性与力约束 | Hierarchical Optimization for WIP Humanoids |
| 最快工程原型 | 底盘速度 + 关节目标 | Mobile ALOHA |

最终建议是：**上层采用模式 2 为主、模式 1 和 3 为辅；模式 4/5 仅保留为同平台基线、日志或低层训练标签。** 这是目前论文中最清晰地兼顾移动范围、底盘-上身一体协同、末端精度和跨平台扩展性的接口划分。
